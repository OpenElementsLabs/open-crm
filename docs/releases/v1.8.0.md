# Open CRM v1.8.0 – Release Notes

**Released:** 2026-06-19
**GitHub release:** [v1.8.0](https://github.com/OpenElementsLabs/open-crm/releases/tag/v1.8.0)
**Previous version:** [v1.7.0](https://github.com/OpenElementsLabs/open-crm/releases/tag/v1.7.0) (2026-06-19)

This is a small, focused release that **hardens the Brevo synchronisation**.
The Brevo sync now keeps the audit log and global search index in step with
imported data, paginates reliably through large Brevo accounts, and logs its
progress in far more detail. There are no user-visible UI changes, no new
services, and no database migrations.

---

## Highlights for Admins and Users

### Brevo sync now updates the audit log and search index

Previously, companies and contacts created or updated by the **Brevo
synchronisation** were written straight to the database, bypassing the events
that normally drive the audit log and the global search index. As a result,
Brevo-imported records could be missing from search until the next backend
restart, and did not appear in the Updates / audit log.

The sync now publishes the same create/update events an ordinary save would, so:

- **Global search** picks up Brevo-imported companies and contacts immediately.
- The **audit log / Updates feed** records Brevo-driven changes like any other.

No action is required — this simply makes Brevo data behave consistently with
data entered through the UI.

### More reliable sync of large Brevo accounts

The Brevo API client now paginates correctly through **all** companies and
contacts, accounting for Brevo's quirk that `/companies` is page-indexed while
`/contacts` is offset-indexed. Safety limits were added so a misbehaving API
response can no longer cause a runaway sync:

- Pagination stops automatically when a page returns only already-seen IDs.
- A hard cap of 1,000 pages aborts the sync with a clear error rather than
  looping forever.
- Waiting on Brevo rate limits is capped so the sync cannot hang indefinitely.

### Clearer sync logs

Each sync run now logs page-by-page progress (page number, page size, newly
imported count, running total) and overall timing. This makes it much easier to
diagnose a slow or partial Brevo import from the backend logs.

---

## Other changes under the hood

These changes are not directly visible to end users:

- **Search index bootstrap steps are now read-only transactional.** The
  per-entity bootstrap steps (companies, contacts, comments) run inside a
  `@Transactional(readOnly = true)` context, ensuring lazy associations load
  correctly during the startup re-index.
- **Mock OAuth2 configuration** for local development was adjusted slightly.
- The **v1.7.0 release notes** were added to `docs/releases/`.

---

## Upgrade notes

- **No database migrations** are required for this update. All schemas are
  unchanged from v1.7.0.
- **No new services or environment variables.** The Meilisearch service and all
  configuration introduced in v1.7.0 remain unchanged.
- Configuration (SSO, Brevo, translation API, search) remains fully compatible.
- No data-loss risk. The release is safe to deploy by replacing the backend
  image.
