# Open CRM v1.9.0 – Release Notes

**Released:** 2026-06-26
**GitHub release:** [v1.9.0](https://github.com/OpenElementsLabs/open-crm/releases/tag/v1.9.0)
**Previous version:** [v1.8.0](https://github.com/OpenElementsLabs/open-crm/releases/tag/v1.8.0) (2026-06-19)

This release adds two consumer-facing capabilities — a read-only **MCP server**
that lets AI assistants query the CRM, and **vCard export** for contacts — and
aligns the user model with `spring-services` 1.1.0. It includes **one database
migration** (`V34`) and new **optional** configuration. The MCP server is
**disabled by default**.

Operators should read the **Upgrade notes** before deploying: the user-table
migration runs automatically, and enabling the MCP server has data-protection
implications.

---

## Highlights for Admins and Users

### Export contacts as vCard

Contacts can now be exported as **vCard 3.0 (`.vcf`)** files, including their
photo:

- **Single contact** — export one contact's vCard from its detail page.
- **Bulk export** — export all contacts matching the current filters (search,
  company, tag, Brevo, language) in one `.vcf` file.

The exported vCards can be imported directly into address books (macOS/iOS
Contacts, Google Contacts, Outlook, …). Existing data is untouched — this is a
read/export feature only.

### MCP server for AI assistants (disabled by default)

Open CRM can now expose a **read-only [Model Context Protocol](https://modelcontextprotocol.io)
(MCP) endpoint at `/mcp`**, so a self-hosted AI assistant such as
[Onyx AI](https://onyx.app) can search and read CRM data live — companies,
contacts, tasks, tags, comments, global search, and a reduced users view.

Important characteristics:

- **Off by default.** The endpoint only exists when `MCP_ENABLED=true`.
- **Read-only.** No tool can create, change, or delete data.
- **Authenticated.** In this release (Phase 1) access is gated by an existing
  **API key** (`X-API-Key`). Note that enabling MCP grants **every existing API
  key** read access to all CRM data via MCP — review your API keys before
  enabling.
- **Paginated.** Results are page-based (default 20, max 50 per page).
- **Audited.** Each tool call is recorded as an access-log entry.

⚠️ **Do not enable the MCP server in production** before completing the GDPR /
data-protection checklist documented in
`docs/specs/108-mcp-connector/design.md`. Backups and live MCP access both
expose personal data and must be governed accordingly. Setup instructions for
Onyx are in the README under **"MCP Server (Onyx AI)"**.

## Other changes under the hood

These changes are not directly visible to end users:

- **`spring-services` upgraded to 1.2.0** (from 1.0.0). This brings the
  expanded user model (see below) and the MCP support classes
  (`com.openelements.spring.base.mcp.*`).
- **User model aligned with `spring-services` 1.1.0.** The `users` table gains
  `external_id` (stable IdP identifier), `user_name` (preferred username),
  and `active` columns; the `sub` column becomes nullable so non-JWT users no
  longer require it. See the migration note below.
- **Search service extracted.** Search logic moved out of `SearchController`
  into a dedicated `CrmSearchService`, which the new MCP tools reuse.
- **Repository reorganisation.** The `specs/` directory was moved to
  `docs/specs/`. No product behaviour is affected.

## Upgrade notes

### Database migration

One Flyway migration runs automatically on startup:

- **`V34`** – aligns the `users` table with `spring-services` 1.1.0. It makes
  `sub` nullable; adds `external_id` (nullable, unique), `user_name`
  (NOT NULL, unique), and `active` (NOT NULL, default `true`); and backfills
  `user_name` from `sub` for existing rows before applying the NOT NULL and
  unique constraints.

No data loss. The backfill makes the migration safe for existing user rows. The
migration is fast on typical CRM-sized data sets.

### New optional configuration (MCP)

All MCP configuration is **optional** and the server is **off by default**.
New environment variables:

- `MCP_ENABLED` – default `false`. Master switch for the `/mcp` endpoint.
- `MCP_AUTH_API_KEY_ENABLED` – default `true`. Enables API-key (`X-API-Key`)
  authentication for MCP (Phase 1).

The provided `docker-compose.yml` and `.env.example` already include these with
safe defaults. A new `openelements.mcp.*` Spring config block exposes server
name/version and pagination limits (default page size 20, max 50). If you do not
set `MCP_ENABLED=true`, nothing changes from v1.8.0.

> The `MCP_AUTH_OIDC_ENABLED` switch (Phase 2 — per-user Claude access via
> Authentik OIDC) is present but **not** wired up for general use in this
> release; leave it at its `false` default.

### No new required services

Unlike v1.7.0, this release adds **no** new infrastructure. Meilisearch and the
rest of the stack are unchanged. The MCP server runs inside the existing
backend.

### Configuration / compatibility

- Existing SSO, Brevo, translation, and search configuration remain fully
  compatible.
- No reverse-proxy changes are required unless you choose to expose `/mcp`
  externally for an AI assistant.
